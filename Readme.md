# 🖨️ Containerized Print Pipeline (CUPS + Docker)

## 🎯 Why this project exists

Printing systems sound simple… until you actually need to build one.

In real environments, printing quickly becomes a coordination problem:

* Files are uploaded from different sources
* Users expect reliable and traceable print jobs
* Printers may be local, remote, or virtual
* Failures are hard to debug (did the file fail? the printer? the network?)
* There is often **no clear visibility** of what happened to a document

This project was built to explore and solve that problem:

> **How do we design a reliable, observable, and extensible print workflow system using modern tools?**

---

## 🧠 What this system does

This is a **containerized print orchestration system** built around CUPS.

It introduces structure where traditional printing lacks it:

* 🧾 Jobs are tracked in a database
* 🔄 Processing is asynchronous (worker-based)
* 🖨️ Printing is handled by CUPS via IPP
* 📄 Output is generated via a virtual PDF printer
* 📡 Everything is containerized and reproducible

## 🔄 How it works (simple view)

```text
User → API → Database → Worker → CUPS → PDF Output
```

### Step-by-step

1. A file is available in the shared storage
2. A print job is created via API
3. The job is stored in PostgreSQL (`QUEUED`)
4. The worker picks it up
5. The file is processed/prepared
6. The job is sent to CUPS
7. CUPS executes printing (PDF backend)
8. Output file is generated


## 🧪 Try it yourself

### 1. Register a printer

```sql
INSERT INTO public.printers (name, cups_uri, status)
VALUES (
    'PDF',
    'ipp://cups_server:631/printers/PDF',
    'ONLINE'
);
```

---

### 2. Submit a job

```bash
curl -X POST http://localhost:8081/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "file_path": "Backend_document.pdf",
    "submitted_by": "test-user",
    "printer_name": "PDF"
}'
```

---

### 3. Observe the system

* The job will move from `QUEUED` → `PROCESSING` → `COMPLETED`
* The worker sends it to CUPS
* A PDF file is generated by the virtual printer

## ⚠️ Current Limitations (MVP Scope)

This project focuses on **core workflow correctness**, not full production hardening.

### 🔐 Security

* No authentication or authorization
* No TLS / mTLS
* No secrets management

### 📊 Observability

* Logs only (no metrics or dashboards)

### 🔁 Reliability

* No retry mechanism
* No dead-letter queue

### 🖥️ Control Plane

* No UI for managing printers or jobs

---

## 🧠 Design Philosophy

* PostgreSQL used as a simple job queue
* Worker is stateless and scalable
* CUPS is treated as execution backend
* Shared volumes simplify integration
